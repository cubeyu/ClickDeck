# 长图导出技术可研 (Task 45A)

## 1. 权限结论
- **当前 manifest 权限是否足够调用 `chrome.tabs.captureVisibleTab`？**
  足够。当前的 `manifest.json` 中已经包含了 `activeTab` 权限以及 `<all_urls>` 的主机权限。`chrome.tabs.captureVisibleTab` 只需要当前标签页的主机权限。因为 ClickDeck 是通过用户主动点击扩展图标唤起的，`activeTab` 权限会被授予，这允许我们在用户当前交互的标签页中进行截图。
- **需要新增权限吗？**
  **不需要新增任何权限**。目前的 `activeTab` 配合 `<all_urls>` 已经完全满足截取可视区域的要求，不需要额外申请如 `tabs` 等更敏感的权限。

## 2. 推荐技术路线与不推荐技术路线

### 推荐技术路线（Content Script 控制流 + Service Worker 截图 + Content Script Canvas 拼接）
由于 `chrome.tabs.captureVisibleTab` 是只能在 Service Worker 中调用的 API，而 DOM 操作（如滚动、隐藏 UI、构建 `<canvas>`）必须在页面上下文中完成。
因此，**最佳方案是两者配合**：
1. **Content Script** 控制页面滚动、隐藏/恢复 UI、修改固定定位元素，并逐屏计算高度向 Service Worker 发起截图请求。
2. **Service Worker** 调用 `captureVisibleTab`，并通过消息回复将 Base64 数据返回给 Content Script。
3. **Content Script** 在内存中创建离线 `<canvas>`，将所有的截图 Data URL 通过 `ctx.drawImage` 按顺序拼接到 Canvas 上，最终通过 `canvas.toDataURL()` 生成整图。

### 不推荐技术路线
- ❌ **纯 Service Worker 拼接**：MV3 中的 Service Worker 无法使用 DOM API，不能创建 `<canvas>`。即使使用 Offscreen Document，也增加了很多额外的消息传递和架构复杂度，不如直接在 Content Script 中拼接简单直观。
- ❌ **纯 Content Script 截图 (html2canvas / dom-to-image)**：这类方案不能真实反映页面的渲染状态（很多 CSS3 特性、跨域图片都会出错），且性能差，不能达到“所见即所得”的高保真截图要求。必须使用浏览器原生的 `captureVisibleTab`。

## 3. MVP 实现拆分
长图导出 MVP 可拆分为以下步骤：
1. **隐藏 UI 与前期准备**：在 `document.documentElement` 添加 `.clickdeck-exporting`，隐藏 ClickDeck 面板、边框等 UI，并保存当前滚动位置。
2. **处理 fixed 元素**（第一版可选）：为了避免长图中固定悬浮头重复出现，可以用简单的 CSS（如针对 fixed 的强制 absolute）或遍历所有 fixed 元素临时隐藏（首屏除外）。
3. **逐屏截图循环**：
   - 计算出 `window.innerHeight` 和 `document.documentElement.scrollHeight`。
   - 每次 `window.scrollTo(0, nextY)` 后，等待 `150-300ms`（保证渲染和加载）。
   - 向 Service Worker 发送截图请求。
   - 收到 Base64 图片后存入数组，直到到达页面底部。
4. **Canvas 拼接**：利用 Content Script 创建总高度的离线 Canvas，循环绘制保存的图片。
5. **恢复现场**：恢复滚动位置、移除隐藏 UI 的 class。
6. **提供下载**：将最终长图的 Data URL 触发下载操作或弹出新标签页展示。

## 4. 清晰度策略
为了保证导出的图片不模糊，必须考虑设备像素比（Device Pixel Ratio）。
`captureVisibleTab` 默认捕获的是**物理像素**级别的截图，即其尺寸为 `(window.innerWidth * window.devicePixelRatio) x (window.innerHeight * window.devicePixelRatio)`。
- 拼接时的 Canvas 尺寸也必须按物理像素来设置：`canvas.width = window.innerWidth * window.devicePixelRatio`，`canvas.height = totalHeight * window.devicePixelRatio`。
- 在 `ctx.drawImage` 绘制时，不需要 CSS 像素转换，直接使用物理坐标逐张拼合，即可实现 1:1 的高保真清晰度。

## 5. 超长页面限制
浏览器对 Canvas 尺寸有严格的硬性限制（例如 Chrome 的最大宽高通常为 32,767px，最大面积约为 2.68 亿像素，即 16,384 × 16,384）。
- 当页面的物理高度（`scrollHeight * devicePixelRatio`）超过浏览器的 Canvas 上限时，长图合成会失败（变成全黑或透明）。
- **处理策略**：
  - 在开始截图前计算预计长图的总物理高度。
  - 如果超限，第一版可直接阻断并提示用户“页面过长，无法生成完整长图”。
  - 后续版本可考虑分片下载（保存为 `Part 1`, `Part 2` 多张图）或者按比例 Downscale（缩放）。

## 6. 与图片版 PDF 的复用关系
长图导出是图片版 PDF 路线的重要前置技术。这两者的前期核心逻辑高度一致。
- **相同点**：都需要通过 Content Script 和 Service Worker 配合实现**逐屏截图采集（Screen Capture Collector）**，并返回一组高质量截图数组。
- **不同点**：
  - 长图导出：拿到图片数组后，画在一个**巨大的 Canvas** 上，导出为 PNG。
  - 图片版 PDF：拿到图片数组后，通过 `jsPDF` 生成文档，每一张截图作为 PDF 的一页，或者将这批长图按 A4 纸比例“切片”后放入 PDF 页中。
- 因此，后续在代码实现时，可将逐屏滚动的“截图收集”引擎抽象为一个通用的模块，后续的长图和图片版 PDF 均能复用该模块。
