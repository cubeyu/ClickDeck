# 图片版 PDF 导出技术可研

## 1. 背景与目标
之前的 HTML/CSS 原生 PDF 打印方案因浏览器、操作系统、打印预览环境差异，存在大量难以低成本抹平的兼容性问题（如背景色失效、分页截断、排版错乱）。
考虑到 PDF 导出的主要用户场景在于“固定化保存与归档阅览”，不要求 PDF 内容可编辑，因此我们转向**图片版 PDF（Image-based PDF）**方案。

本可研旨在明确图片版 PDF 的技术路线，确保能复用稳定的“长图截取”链路，并为三种导出模式（A4、长页、幻灯片）制定清晰的切片与组装规则。

## 2. 技术路线与实现方案
### 2.1 核心链路
完全复用现有的**屏幕滚动与分段截取链路**（Screen Capture Collector）：
1. **触发与 UI 隐藏**：通过 Content Script 给 `documentElement` 添加 `.clickdeck-exporting`，隐藏所有 ClickDeck UI 控制面。
2. **滚动收集**：Content Script 控制页面按一定步长（如 `window.innerHeight` 或计算好的幻灯片高度）滚动。
3. **原生截图**：每次滚动后，等待渲染稳定，通过 Service Worker 调用 `chrome.tabs.captureVisibleTab` 获取无损物理像素截图（Data URL）。
4. **客户端 PDF 组装**：不依赖外部服务端接口。在 Content Script 内存中引入前端 PDF 生成库（如 `jspdf` 或 `pdf-lib`）。
5. **图片切片与绘制**：将截取收集到的图片流，按模式要求计算坐标与切片规则，用 `pdf.addImage()` 绘制到 PDF 页面上，最后触发 `pdf.save()` 进行本地下载。

### 2.2 依赖引入
推荐使用 **jspdf** (`npm install jspdf`)。它在客户端使用 Canvas/图片构建 PDF 方面非常成熟，支持任意自定义页面尺寸。

## 3. 三种 PDF 模式的切片规则

### 3.1 长页模式 (Long-page)
- **目标**：不分页，整个网页就是一张无缝连贯的 PDF。
- **采集逻辑**：按视口高度往下滚动截取，计算得到页面总高。
- **绘制规则**：
  1. 实例化单页 PDF，页面尺寸设为 `[视口总宽度, 网页总高度]`（可根据屏幕 DPR 换算成 PDF 物理单位 mm 或 pt）。
  2. 将收集到的截图片段，按其实际滚动 Y 坐标 `y * dpr` 为起点依次往下贴合，直至覆盖整页。
  3. PDF 文件将仅有 1 页，但长度极长。

### 3.2 A4 模式 (A4)
- **目标**：方便标准 A4 纸张打印浏览。
- **采集逻辑**：与长图模式一致，先拿到所有截图并可在内存中利用隐藏的 Canvas 拼合出完整的超长 Image。
- **绘制规则**：
  1. 实例化 A4 尺寸的 PDF（标准比例：宽 `210mm`，高 `297mm`）。
  2. 根据 A4 比例将拼合后的超长长图进行“硬性水平切割”（切片高度 `H = 图片总宽 * (297 / 210)`）。
  3. 循环创建 PDF 新页，将对应片段 `[0, i*H, 图片总宽, H]` 的区域裁切出来绘制在当前页。
  4. **已知限制**：由于是纯图片切割，跨越切割线的文字或图片会被直接从中间截断。这在不可编辑的归档使用场景中是可容忍的常见情况。

### 3.3 幻灯片模式 (Slides, 16:9)
- **目标**：还原网页原本类似 PPT 的横向翻页体验。
- **采集逻辑**：
  1. 不再单纯按视口高度盲滚。需要先探明每个具有 `.slide` 或对应宽高的模块节点的位置（`offsetTop`）。
  2. 针对每个幻灯片节点，使用 `scrollIntoView()` 滚动对齐，然后请求截图。
- **绘制规则**：
  1. 实例化 16:9 横向（Landscape）尺寸的 PDF（例如 `[1920, 1080]` 换算的物理单位）。
  2. 对每一张截取到的幻灯片图片，为其生成一个新的 PDF 页面。
  3. 将该图片以 cover 满铺（或 contain 居中）的方式画入 PDF 页面中。
  4. 最终 PDF 会与幻灯片张数一致。

## 4. 风险与已知限制
1. **悬浮元素（Fixed Elements）复现**：由于使用的是滚动截图法，页面上未被隐藏的 `position: fixed`（如顶部导航条）会在每次截图中出现。
   - *对策*：与长图导出一样，当前 MVP 阶段予以容忍，视为页面本身的显示特性；后续可提供移除 Fixed 元素的实验选项。
2. **极长网页的内存崩溃**：Canvas 拼装或者超大 base64 字符串塞入 jsPDF 可能引起低配机器的 OOM（Out of Memory）。
   - *对策*：在组装时限制单张图片的最高分辨率上限，或对捕获的屏幕数据做适当缩放（例如在绘制前压缩为 JPEG），平衡清晰度与文件体积。
3. **跨页硬切割**：A4 模式没有 DOM 节点的上下文，所以必定存在物理“切断文字”的现象。
   - *对策*：由于无需编辑，仅需在 UI 提示或者默认策略中说明。

## 5. 必须回答的核心问题答复

1. **长页 PDF 是否可以由长图直接生成？最大高度和文件体积风险是什么？**
   - **可以。** 生成方式是将收集好的长幅 Canvas 图像一次性通过 `addImage` 贴入高度与图片等比的自定义尺寸 PDF 页面中。
   - **风险**：主流 PDF 阅读器对单页最大物理尺寸有上限（如 Acrobat 曾限制 200 英寸），且 jsPDF 内部处理超大 base64 字符串会造成内存突增。极长网页可能会导致浏览器 OOM 或者生成的 PDF 无法被旧版阅读器打开。

2. **A4 图片 PDF 如何切片？按 A4 比例切图，还是先布局再截图？**
   - **按 A4 比例直接对长图切片**。由于我们要保持原有“所见即所得”的布局效果，不应该改变页面的流式布局（重排极易引发错乱）。所以采取先截取完整的原貌长图，再基于纯数学高度（H = W * 297/210）在 Canvas 上逐段切割，贴入多张 A4 尺寸的 PDF 页中。

3. **16:9 图片 PDF 如何处理 slide deck？是每个 slide 截一页，还是长图切片？**
   - **每个 slide 单独对齐截一页**。不能使用长图硬切，因为幻灯片在网页上不一定严格紧贴。正确的做法是利用已有的 `detectPresentationSlides()` 找到幻灯片节点，用 `scrollIntoView()` 居中对齐每一页，单页截图后作为完整的一页放入 16:9 PDF 中。

4. **图片版 PDF 是否需要新增 npm 依赖？如果需要，候选依赖是什么，license 是否可接受？**
   - **需要**。候选依赖为 `jspdf`。
   - **License**：`jspdf` 使用 MIT License，完全可以接受并在商业/开源闭环内自由使用。

5. **图片版 PDF 是否保留文字可选中？如果不保留，界面文案如何说明？**
   - **不保留**。这是纯图片构成的 PDF，文字全部被栅格化，不支持鼠标拖拽选中和复制。
   - **界面文案说明**：在 PDF 导出菜单或 tooltip 处需注明：“PDF (图片版) - 布局 100% 还原，但文字不可复制” 等类似提示。

6. **与旧 `window.print()` PDF 相比，哪些问题会被解决，哪些问题仍然存在？**
   - **会被解决的问题**：背景色无法打印问题、Chrome/Edge 的奇怪分页符失效问题、页面动画导致排版错位问题、打印预览界面干扰、不同电脑打印效果不一致问题。真正做到“所见即所印”。
   - **依然存在的问题**：文字无法选中（降级为图片）；固定定位的吸顶元素在长图流或者 A4 切图时仍会重复出现；超长页面 OOM 风险。

7. **是否应该保留旧 PDF 按钮，还是隐藏/标记为实验功能？**
   - **建议彻底隐藏**。因为旧版 PDF 存在难以修复的浏览器引擎差异，保留该入口只会增加使用者的迷惑和困扰，并且这与当前“固定化保存与归档阅览”的确定性方向不符。

## 6. 结论
图片版 PDF 具备高度的还原确定性，避免了浏览器底层打印系统的黑盒怪异行为。以复用 `captureVisibleTab` 的截图收集器作为基座，配合 `jspdf` 负责排版切片，是完全可行且稳妥的交付方案。不需要改变既有架构或权限。
